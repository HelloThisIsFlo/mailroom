Fix step 1 in apply_reset() (lines 286-293 of resetter.py):

CURRENT (broken): For each label, just calls `batch_remove_labels(email_ids, [mb_id])`. When email is only in that mailbox, this violates RFC 8621 (empty mailboxIds).

FIX: For each managed label in step 1:
1. First call `jmap.batch_add_labels(email_ids, [screener_mb_id])` to ensure every email has Screener as a fallback mailbox
2. Then call `jmap.batch_remove_labels(email_ids, [mb_id])` to remove the managed label

This is an atomic MOVE pattern: add destination first, then remove source. Semantically correct: reset undoes triage, emails go back to where they were before triage.

IMPORTANT: The Screener mailbox is NOT a triage trigger — triage is triggered by triage labels (e.g., @Feed, @Paper Trail). Moving emails to Screener without a triage label will NOT cause re-triage on next poll. Verified via code analysis of ScreenerWorkflow.poll() which queries by triage label, not by Screener mailbox.

Also ensure the screener_mailbox name (from settings.triage.screener_mailbox) is included in the `resolve_names` list (line 277) so screener_mb_id is available. Currently resolve_names only includes label_names + warning + error. Add screener_mailbox to the set.

Note: batch_add_labels and batch_remove_labels already chunk in BATCH_SIZE=100, so large volumes (1000+ emails) are handled correctly.

Update affected tests:
- test_step1_removes_managed_labels: Now also asserts batch_add_labels called with Screener ID before batch_remove_labels
- test_7step_operation_order: Update call_order tracking to include the new batch_add_labels calls in step 1
- test_result_counts: May need adjustment if the add-Screener calls change any mock behavior

Per research: RFC 8621 Section 4.1 requires emails belong to at least one mailbox. Confirmed via live JMAP test against Fastmail — removing last mailbox returns invalidProperties error. Emails filed to Feed/Jail/Paper Trail without add_to_inbox have ONLY that destination mailbox. Removing it = empty mailboxIds = Fastmail rejects.
