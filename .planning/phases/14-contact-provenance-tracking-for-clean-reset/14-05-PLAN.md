1. **Config error simplification** (src/mailroom/core/config.py):
   Change the `reject_old_labels_key` validator message from:
   `"Config section 'labels:' has been renamed to 'mailroom:'. Please update your config.yaml."`
   to something like:
   `"Unknown configuration key 'labels'. Valid top-level keys: triage, mailroom, logging."`
   This is a plain rejection without backward-compatibility guidance, per user feedback "should just reject as unknown key."
   Update the test in tests/test_config.py that asserts on this error message.

2. **REV field unit test** (tests/test_resetter.py):
   Add `test_rev_field_alone_returns_true` to TestIsUserModified class:
   - Build a vCard with standard Mailroom fields + `REV:2026-03-04T12:00:00Z\r\n`
   - Assert `_is_user_modified(vcard) is True`
   This documents that REV (added by Fastmail on every contact edit) is NOT in MAILROOM_MANAGED_FIELDS and therefore triggers user-modification detection. This is the key mechanism that makes the whole "user-modified" concept work with Fastmail's CardDAV server.

3. **REV documentation** (src/mailroom/reset/resetter.py):
   Add a comment block above or below MAILROOM_MANAGED_FIELDS explaining:
   - REV is intentionally NOT in this set
   - Fastmail adds/updates REV timestamp on every contact edit via their UI
   - This means any user edit produces a REV field, which triggers _is_user_modified()
   - This is a relied-upon behavior of Fastmail's CardDAV implementation
   - If the CardDAV server does NOT add REV on edit, user-modification detection would need a different strategy
